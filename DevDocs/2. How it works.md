# How Pezhvak Works

This document details the internal technical processes of Pezhvak Core, from the moment a user hits "Send" to the moment a message is relayed or decrypted by a peer.

## 1. The Message Lifecycle (Sending)

When your application calls `SendPlaintextMessage(peerID, recipientPubKey, plaintext)`, the following sequence occurs:

1.  **Encryption:** The Core uses NaCl `box` (Curve25519 + XSalsa20 + Poly1305) to encrypt the plaintext using your private key and the recipient's public key.
2.  **Encapsulation:** The encrypted data is wrapped in a `PezhvakMessage` Protobuf object, which includes the `SenderId`, `RecipientId`, and a `Timestamp`.
3.  **Fragmentation:** Because BLE MTU sizes are small (often < 512 bytes), the Core breaks the serialized Protobuf message into chunks based on the current **Power Level** (`Normal`, `High`, or `Max`).
4.  **Transmission:** Each chunk is wrapped in a `BLEPacket` (containing `MessageId`, `ChunkIndex`, and `TotalChunks`) and sent via the `NativePlatform.SendBLE` interface.

## 2. Packet Reassembly (Receiving)

When the native radio receives raw bytes, it passes them to the Core via `ReceiveFromBLE(peerID, rawPacket)`.

1.  **The Router:** The `Router` tracks incoming `BLEPacket` objects using an internal `messageAssembler`. 
2.  **Deduplication:** Before processing, the Core checks the `MessageID` against a "seen" index in BadgerDB. If the message was already processed in the last 72 hours, it is discarded immediately to save CPU and battery.
3.  **Buffer Management:** To prevent memory exhaustion (OOM), the Router limits message sizes to 5,000 chunks (~1MB). If chunks don't arrive within 60 seconds, the partial message is wiped from RAM.

## 3. The "Mule" Logic (Relaying vs. Decrypting)

Once a message is fully reassembled, the Core inspects the `RecipientId` (which is outside the encrypted "box"):

*   **Scenario A: It's for us.**
    *   The Core attempts to decrypt the payload using the `SenderId` provided.
    *   On success, the plaintext is passed up to the UI via `OnMessageReceived`.
*   **Scenario B: It's for someone else (Mesh Relay).**
    *   The Core acts as a "Mule." It stores the encrypted payload in the `pending` table of BadgerDB.
    *   The message remains there for up to 48 hours (TTL).
    *   Whenever the Core connects to a new peer, it calls `SyncPendingMessages`. If that peer is the intended recipient (or another potential relay), the Core fragments and sends the stored message.

## 4. Power Profiles & Performance

The performance of the processes above is governed by the `SetRadioPowerLevel` API:

| Level | Payload Size | Inter-packet Delay | Native Behavior |
| :--- | :--- | :--- | :--- |
| **Normal** | 200 bytes | 20ms | Low TX Power, Slow Scan |
| **High** | 450 bytes | 10ms | High TX Power, Balanced Scan |
| **Max** | 480 bytes | 0ms | Max TX Power, Continuous Scan |

## 5. Persistence and Security

### Data Storage
Pezhvak uses **BadgerDB**, a high-performance Key-Value store. It manages three primary data types:
1.  **Pending:** Messages waiting for delivery to a peer (`pending:peerID:msgID`).
2.  **Seen:** A rolling index of message IDs to prevent loops (`seen:msgID`).
3.  **Identity:** Your unique Curve25519 keypair.

### The Panic Wipe
When `WipeAllData()` is triggered:
1.  The database connection is paused.
2.  `db.DropAll()` is called, physically removing all SSTables from the disk.
3.  The `identity.json` file should be deleted by the UI layer (The Core provides the primitive, the UI handles the file-system cleanup).
