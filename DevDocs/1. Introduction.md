# Introduction to Pezhvak Core

Welcome to the Pezhvak Core developer documentation. Pezhvak Core is a specialized "engine" for decentralized, resilient, and secure communication. It is designed to operate in environments where traditional internet infrastructure is unavailable, compromised, or restricted.

## What is Pezhvak Core?

At its heart, Pezhvak is a **Store-and-Forward Mesh Networking Engine**. Unlike traditional chat apps that require a central server, Pezhvak nodes communicate directly with one another over Bluetooth Low Energy (BLE). 

If a recipient is out of range, any other node in the network can act as a "carrier," storing the encrypted message and passing it along when they encounter the recipient or another relay.

## The Core Architecture

To remain cross-platform, Pezhvak Core is written in Go and exposes a clean interface via `gomobile` (for Android/iOS) and C-Shared libraries (for Desktop).

### 1. The Integration Model
Integration follows a "Plugin" pattern. Your application provides the **Transport Layer** (handling the physical Bluetooth radio) and the **UI**, while the Core handles:
*   **Cryptography:** Automatic End-to-End Encryption (E2EE) using NaCl `box`.
*   **Fragmentation:** Breaking large messages into BLE-safe packets and reassembling them.
*   **Persistence:** Managing the `BadgerDB` message store and deduplication.
*   **Mesh Logic:** Deciding whether to decrypt a message (if it's for you) or carry it (if it's for someone else).

### 2. Key Concepts

#### Identities
Every user is identified by a **Curve25519 Public Key**. There are no usernames or phone numbers stored in the core. If a user loses their `identity.json`, they lose their identity.

#### Message Retention (TTL)
By default, the Core acts as a "Mule" for **72 hours (3 days)**. If a message cannot be delivered to its recipient or another relay within this window, it is automatically purged from the local BadgerDB store to save space.

#### Power Levels
To balance battery life and performance, the Core supports three distinct power profiles that your GUI should expose to the user:
*   **Normal:** Optimized for background battery life.
*   **High:** Increases transmission power and packet size for active use.
*   **Max (Turbo):** Full-throttle throughput and continuous scanning for emergency or high-speed data transfer.

#### The "Panic Button"
In high-risk scenarios, the Core provides a `WipeAllData()` function. When called, it instantly drops all database tables and clears identity keys, leaving no trace of communication history on the device.
