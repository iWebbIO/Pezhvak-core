# Integration Guide (Part 2): Transport & Sync

Once the Core is initialized and your `NativePlatform` bridge is ready, the next step is to manage the lifecycle of Bluetooth connections and data packets.

## 1. Handling Incoming Data

When your native BLE stack receives data on the **Pezhvak RX Characteristic** (usually via a "Notify" or "Indication" callback), you must pass that raw byte array directly to the Core.

The `ReceiveFromBLE` method handles the reassembly of chunks. You do not need to worry about packet ordering or fragmentation at the UI level.

### Android (Kotlin)
```kotlin
// Inside your BluetoothGattCallback
override fun onCharacteristicChanged(gatt: BluetoothGatt, characteristic: BluetoothGattCharacteristic) {
    if (characteristic.uuid == PEZHVAK_RX_UUID) {
        val rawData = characteristic.value
        val peerID = gatt.device.address // Use MAC address or unique UUID
        
        try {
            pezhvak.receiveFromBLE(peerID, rawData)
        } catch (e: Exception) {
            Log.e("Pezhvak", "Router error: ${e.message}")
        }
    }
}
```

### iOS (Swift)
```swift
// Inside your CBPeripheralDelegate
func peripheral(_ peripheral: CBPeripheral, didUpdateValueFor characteristic: CBCharacteristic, error: Error?) {
    guard let data = characteristic.value, characteristic.uuid == PEZHVAK_RX_UUID else { return }
    let peerID = peripheral.identifier.uuidString
    
    do {
        try pezhvak.receiveFromBLE(peerID, data: data)
    } catch {
        print("Router error: \(error.localizedDescription)")
    }
}
```

## 2. Synchronizing Stored Messages

This is a key part of the Pezhvak mesh. Because nodes act as carriers (mules), your device might be holding encrypted messages for a peer you just encountered. 

**You must call `SyncPendingMessages` immediately after a successful BLE connection is established.**

### When to call Sync:
1.  As a **Central**: After `connectGatt` succeeds and services are discovered.
2.  As a **Peripheral**: After a remote Central subscribes to your Notify characteristic.

```kotlin
fun onPeerConnected(peerID: String) {
    // This triggers the Core to check BadgerDB for any messages 
    // destined for this specific peer (or meant to be relayed by them).
    pezhvak.syncPendingMessages(peerID)
}
```

## 3. Sending a New Message

Sending a message is a high-level operation. You provide the plaintext and the **Recipient's Public Key**. The Core handles the encryption and the fragmentation into chunks.

```kotlin
val recipientKey = "a1b2c3d4..." // Hex string from your contact list
val message = "Hello, stay safe!".toByteArray(Charsets.UTF_8)

try {
    // peerID is the identifier for the current active BLE connection
    pezhvak.sendPlaintextMessage(peerID, recipientKey, message)
} catch (e: Exception) {
    // If the BLE transmission fails, the Core automatically saves 
    // the message to the 'pending' store for a later retry/sync.
    Log.d("Pezhvak", "Message queued for later: ${e.message}")
}
```

## 4. Best Practices for Reliability

*   **MTU Negotiation:** For optimal performance (especially in `High` or `Max` power modes), try to negotiate a higher MTU (up to 512 bytes) on the native side. The Core will automatically utilize the available space based on the `payloadSize` setting.
*   **Peer IDs:** Keep your `peerID` strings consistent during a single session. The Core uses this ID to map incoming chunks to the correct reassembly buffer. If the ID changes mid-stream, the message reassembly will time out (60s).
*   **Queue Management:** `SyncPendingMessages` is non-blocking in Go, but since it involves database I/O and BLE writes, avoid calling it inside a tight loop or on the main UI thread if possible.
